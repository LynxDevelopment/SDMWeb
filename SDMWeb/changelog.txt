XWEB 4.0.2

- aggiunta del template manager:
	Questo sistema permette di definire ed applicare template di pagine che includeranno i contenuti richiesti.
	Le pagine dell'applicazione sono ridotte all'essenziale, delegando ai template le parti ripetute.

- interfaccia rinnovata:
	l'interfaccia di amministrazione è stata rinnovata utilizzando un foglio di stile css più sofisticato e generale e un template applicato attraverso il template manager. Lo sviluppo di nuove pagine di amministrazione custom è ora più semplice e veloce.


XWEB 4.0.1

- cambiato url mapping delle action da "/do?action=<action_name>" a "<action_name>.do".
	Con il nuovo mapping è possibile dare alle action un nome del tutto identico ad un path e richiamare la action in modo assoluto oppure relativo come se fosse un file.
	Inoltre è ora possibile proteggere le action attraverso i security constraint definiti in J2EE con granularità qualsiasi.

- modificato sistema di tracking:
	il tag "tracking" da inserire nelle pagine perchè ne sia tracciato l'accesso è stato sostituito da un filtro (Filter come definito in J2EE) che attiva il sistema di tracking ad ogni accesso alle risorse su cui è mappato.
	Il nuovo sistema di tracking consente di accedere alle informazioni relative alle sessioni attive e di eseguire dump su database periodici o su richiesta di tutte le informazioni.


- nuovo sistema di gestione dei menù:
	attraverso un nuovo sistema di gestione dei menù si sono ottenute migliori performance, maggiore robustezza e semplicità d'uso. La voce di menu selezionata viene mantenuta in sessione cosicchè l'intervento manuale per modificarla si rende necessario solo in link realizzati manualmente, cioè non utilizzando gli appositi tag.

- introduzione dei componenti dataset:
	attraverso questi componenti e la tag library dedicata si è fornito un sistema alternativo all'utilizzo di XML + XSL per gestire dati tabellari provenienti da un database.

- cambiata gestione delle anagrafiche "custom":
	la realizzazione di classi di utenti custom è stata semplificata attraverso l'introduzione di un'apposita struttura dati nel database piuttosto che attraverso la configurazione come avveniva precedentemente.

- introduzione del sistema di inizializzazione:
	il sistema di inizializzazione permette di costruire componenti che vengono inizializzati e terminati all'avvio e allo shutdown di xweb. Questo sistema gestisce il file di configurazione e il logging delle eccezioni in maniera che la configurazione di xweb risulti maggiormente assistita e in maniera da renderla riusabile per eventuali componenti custom.
	Tale sistema permette inoltre di riavviare (senza che vengano perse le sessioni utente) un qualsiasi componente oppure xweb interamente attraverso una chiamata http o nel momento in cui il file di configurazione venga modificato (auto-reload).

- auto-reload della configurazione delle action:
	i file di configurazione delle action possono ora essere in numero qualsiasi (configurati all'interno del file xweb_config.xml) e per ciascuno può essere attivato il sistema di auto-reload che permette di ricaricare la configurazione modificata senza riavviare nè xweb, nè l'application server.

- cambiato metodo di invio della pagina di uscita delle action:
	cambiando il metodo di ridirezione dalla action alla pagina successiva da forward a redirect si è risolto il problema della ripetizione dell'esecuzione della action al refresh della pagina (ove questo effetto è stato ritenuto indesiderato).
